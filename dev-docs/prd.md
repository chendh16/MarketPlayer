1. 背景与目标
背景
华人个人投资者普遍同时参与A股、港股、美股、BTC多个市场，面临三大核心困境：信息量远超个人处理能力、跨时区导致频繁错过交易时机、多平台分散操作效率低下。现有工具要么只覆盖单一市场，要么只提供信息不能执行，没有一个产品能完整打通"信息→决策→执行→风控"的全链路。
目标
打造一个面向华人个人投资者的全市场AI智能交易助手，核心实现四件事：
●7×24自动监控四大市场，主动推送关键信号，解决错过交易时机的问题
●AI生成中文交易信号参考，用户人工确认后一键下单，解决信息处理效率问题
●内置仓位风控引擎，结合实时账户持仓检查，避免过度持仓引发风险
●全流程异常处理与补救机制，保障交易执行可靠性
产品愿景
让每个华人个人投资者，都拥有一个7×24在线的全市场AI专属分析师和有责任感的交易执行助手。
合规声明策略
本系统提供的所有内容仅为信息参考，不构成任何投资建议。产品所有对外措辞统一使用"信号参考"而非"投资建议"。用户注册时须签署风险披露协议方可使用。启动前需咨询香港/新加坡金融合规律师，明确持牌要求。

4. 用户故事
模块一：资讯与信号
US-01 作为一个无法全天盯盘的投资者，我希望在任意市场出现重要信号时自动收到Discord通知，这样我就不会因为没在看盘而错过机会。
US-02 作为同时关注四个市场的投资者，我希望收到的资讯是中文摘要而不是原文，这样我能快速判断是否值得关注，不需要花时间翻译和消化。
US-03 作为美股投资者，我希望在睡觉时如果美股出现重大事件，系统能在5-10分钟内推送并告诉我信号方向，让我决定是否操作。
US-04 作为投资者，我希望在非交易时段收到的推送有明确标注，让我知道这只是信息参考，开盘后需要重新评估行情再决策。
模块二：AI信号参考
US-05 作为投资者，我希望收到的不只是资讯，而是系统告诉我"看多还是看空、参考仓位多少"，这样我能更快做决策而不是自己从头分析。
US-06 作为风险意识较强的投资者，我希望在看到交易参考的同时，能看到当前持仓状况和风控检查结果，在充分了解风险后再决定是否确认。
US-07 作为投资者，我希望建议有明确的有效期，超时后系统能提醒我，避免我在行情已变的情况下执行过时的参考。
模块三：风控与仓位管理
US-08 作为投资者，我希望系统在给出下单参考前，自动检查我的当前仓位，如果某只股票我已持有太多，系统应该提醒我甚至拦截，避免过度集中持仓。
US-09 作为保守型投资者，我希望能设置自己的仓位上限规则，让系统按我的风险偏好来过滤参考。
US-10 作为投资者，我希望即使系统发出风险警告，最终决定权仍然在我手上，我可以选择忽略警告，但系统会记录这个决策。
US-11 作为风控拦截后的投资者，我希望系统不只是告诉我"不能买"，还能告诉我需要减仓多少、有哪些替代标的，让我有路可走。
US-12 作为同时使用多个平台的投资者，我希望能手动填写其他平台的持仓数据，让风控计算更准确，而不是只看富途一个平台。
模块四：交易执行
US-13 作为富途用户，我希望在Discord上点击确认后，系统能直接帮我在富途下单，节省时间也减少失误。
US-14 作为投资者，我希望下单失败时系统能告诉我具体原因，并提供明确的补救选项，而不是只显示一条失败通知让我不知所措。
US-15 作为同时使用富途和Longbridge的投资者，我希望系统能智能判断在哪个平台下单，或者让我手动选择。
US-16 作为A股投资者，我希望收到A股信号参考时，系统能帮我复制交易信息，让我能快速前往券商App手动执行。
模块五：账户与设置
US-17 作为新用户，我希望能安全绑定富途账户，整个绑定过程清晰透明，明确知道系统获取了哪些权限。
US-18 作为投资者，我希望能设置风险偏好，系统自动配置对应仓位参数，不需要我手动设置每个数字。

安全

附录D：技术约束
系统架构设计原则
1. 分布式事务与最终一致性 跨平台交易采用事件驱动架构和最终一致性机制，确保多平台数据最终一致，同时保证同步延迟最小化。单次操作失败时通过补偿事务（Compensating Transaction）恢复一致状态。
2. API限流与缓存优化 使用Redis缓存机制和API限流减少高频调用的资源消耗，确保系统在市场波动较大时平稳运行。持仓数据缓存60秒，下单前强制实时二次拉取。
3. 高可用性设计 采用负载均衡和微服务架构，支持高并发、高可用要求。通过自动重试机制和富途连接进程池化确保系统自动恢复故障。目标月可用率＞99%。
4. 下单幂等性保障 每条交易参考生成唯一OrderToken，基于Redis分布式锁实现用户维度的并发控制，确保同一Token只处理一次，从根本上防止重复下单。
5. Discord 3秒交互限制处理 Discord要求按钮交互必须在3秒内响应，否则显示"交互失败"。解决方案：收到交互事件后立即ACK（deferReply），再异步执行风控检查和下单逻辑，最后通过editReply更新消息内容。
6. 富途长连接管理 富途OpenAPI基于TCP长连接，非标准REST API，服务端部署需要独立连接管理进程。每个用户对应独立连接实例，连接断开后自动重连，重连失败时降级至方案B/C并通知用户。
